No existing table is retrofitted (that is A6b/A6c). `e10` is **never** added to PostgREST's exposed schema list (`supabase/config.toml` `[api] schemas` stays `["public","graphql_public"]`).

### Why squashed
The live ledger held 50 migrations; the repo had 13 files, and the ~37-migration gap is **interleaved** (obs/P0 hotfixes are newer than some repo files), so no clean partial prefix exists. Full squash is the only faithful option. **At A1 the squash was repo-side only** ‚Äî the production `supabase_migrations.schema_migrations` ledger was left untouched and nothing was re-applied to prod (A1 changed zero prod state).

**At A4 (first pipeline pass) the production ledger WAS reconciled** to adopt the squashed baseline, so `supabase db push` became the prod mechanism. Reconciliation is metadata-only (no schema/data change): the 50 real pre-squash versions were marked `reverted` (`supabase migration repair --status reverted ‚Ä¶`) and the squashed baseline set (`00000000000000`, `‚Ä¶01`, `‚Ä¶02`, `20260715160000`) marked `applied`, leaving the ledger = {baseline set + P0 hotfix} so `db push` applies only genuinely new migrations. The real historical migration files are preserved in `supabase/migrations_archive/`. From A4 on, the prod ledger tracks the repo (see `docs/OPERATIONS.md` ‚Üí Release runbook).

**Dispositioned difference ‚Äî `pg_net`:** the Supabase local stack installs the `pg_net` extension by default; this production project does not enable it. It is unused by the app and has zero impact on the application schema. Not added to prod (that would be an out-of-scope prod change); not dropped from local (it is Supabase-managed). Accepted.

All other objects ‚Äî tables, functions, views, policies, grants, publications, replica identities ‚Äî match byte-for-byte after comment/blank normalization.

## Local workflow
```bash
cd ~/dev/element10-app
colima start                       # Docker runtime (once)
supabase start                     # local stack (analytics disabled in config.toml ‚Äî colima socket-mount)
supabase db reset                  # re-apply migrations + seed
# regenerate the diff:
supabase db dump --local -f /tmp/local.sql
supabase db dump -p "$SUPABASE_DB_PASSWORD" -f /tmp/prod.sql   # prod (read-only)
diff <(grep -vE '^\s*--|^\s*$' /tmp/local.sql) <(grep -vE '^\s*--|^\s*$' /tmp/prod.sql)   # expect: only pg_net
```
Access (`SUPABASE_ACCESS_TOKEN`, `SUPABASE_DB_PASSWORD`) lives in `.env.local` (gitignored). The prod connection is used read-only, via the IPv4 pooler (`aws-0-us-east-1.pooler.supabase.com`, user `postgres.<ref>`).
