# Element 10 — Database blueprint (canonical)

**Production project:** `ddhkkumiyidorzmajwde` (PostgreSQL 17.6). **The repo's `supabase/migrations/` is now the canonical source of truth for the schema** — production can be recreated from it. (`supabase/schema.sql` is retained as annotated history; it is NOT the reproduction source.)

## Layout
- `supabase/migrations/00000000000000_e10_prod_baseline.sql` — **squashed baseline**: the full live public-schema state (tables, functions, views w/ `security_invoker`, RLS policies, grants/revokes, indexes, extensions, `supabase_realtime` publication membership, replica identities). Generated by `supabase db dump` (containerized pg17).
- `00000000000001_e10_storage_cards_bucket.sql` — the `cards` storage bucket + object policies (the `storage` schema is not in a public-schema dump).
- `00000000000002_e10_baseline_grant_fixups.sql` — revokes the broad table grants (`REFERENCES/TRIGGER/TRUNCATE/MAINTAIN`) that Supabase-local default privileges re-add but production had revoked (see disposition below).
- `20260715130126_e10_p0_recon_view_security.sql` — the P0 recon-view security hotfix, **verbatim** (see `docs/incidents/2026-07-15-m4-blob-clobber.md`).
- `supabase/migrations_archive/` — the 13 pre-baseline files, preserved verbatim (moved out of the apply path, never rewritten).
- `supabase/seed.sql` — minimal LOCAL/STAGING data fixtures (never schema). The 57k-card catalog + 176 rosters + test users load separately (documented in the seed header).

### Why squashed
The live ledger held 50 migrations; the repo had 13 files, and the ~37-migration gap is **interleaved** (obs/P0 hotfixes are newer than some repo files), so no clean partial prefix exists. Full squash is the only faithful option. The production ledger (`supabase_migrations.schema_migrations`) is left untouched — this is a repo-side reorganization; nothing was re-applied to prod.

## Reproducibility proof (A1 acceptance)
`supabase db reset` on an empty local stack applies all four migrations cleanly, and a normalized `pg_dump --schema-only` diff of **local vs production** is empty except for one dispositioned line:

**Dispositioned difference — `pg_net`:** the Supabase local stack installs the `pg_net` extension by default; this production project does not enable it. It is unused by the app and has zero impact on the application schema. Not added to prod (that would be an out-of-scope prod change); not dropped from local (it is Supabase-managed). Accepted.

All other objects — tables, functions, views, policies, grants, publications, replica identities — match byte-for-byte after comment/blank normalization.

## Local workflow
```bash
cd ~/dev/element10-app
colima start                       # Docker runtime (once)
supabase start                     # local stack (analytics disabled in config.toml — colima socket-mount)
supabase db reset                  # re-apply migrations + seed
# regenerate the diff:
supabase db dump --local -f /tmp/local.sql
supabase db dump -p "$SUPABASE_DB_PASSWORD" -f /tmp/prod.sql   # prod (read-only)
diff <(grep -vE '^\s*--|^\s*$' /tmp/local.sql) <(grep -vE '^\s*--|^\s*$' /tmp/prod.sql)   # expect: only pg_net
```
Access (`SUPABASE_ACCESS_TOKEN`, `SUPABASE_DB_PASSWORD`) lives in `.env.local` (gitignored). The prod connection is used read-only, via the IPv4 pooler (`aws-0-us-east-1.pooler.supabase.com`, user `postgres.<ref>`).
